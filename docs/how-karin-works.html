How Karin works

A local viewer for your AI sessions. Three different AI tools write logs on this machine; Karin reads them, converts each into one shared shape, groups them into turns, prices them, and draws them. Nothing leaves the PC.

Codex / Claude / Warp logs→ Python indexers→ JSON feeds in data/→ adapters → UnifiedSession→ cycles→ pricing→ React UI

1Where the data comes from

Each AI tool logs to its own place in its own format. One small Python program per tool reads that and writes a plain JSON file into data/. That is the only code that knows anything tool-specific about reading.

SourceIndexerReadsWrites
Codexbin/karin.py~/.codex/sessions/**.jsonldata/karin-data.json
Claudebin/karin_claude.py~/.claude/projects/**.jsonldata/claude-raw.json

All three redact secrets on the way through, and all three can run with --watch: they poll for changes and rewrite the feed, which is how the page stays current while you work.

Warp is the awkward one. Its per-task log is an undocumented binary blob with no published schema, so the indexer walks the raw format generically and identifies things by field number. It works, but a Warp update could move those numbers. Anything it can't name is still kept verbatim and shows up in the Raw tab, so nothing is silently dropped.

2One shape for all three

This is the central design decision: there is no per-AI user interface. Each feed is converted once, in src/lib/adapt.ts, into a single type called UnifiedSession (defined in src/types.ts). Everything downstream — list, detail, timeline, summary, search, export — only ever sees that type.

A UnifiedSession carries: a unique id, which tool it came from, a title, the working directory, the model(s) used, start/update times, counts of prompts and tools, its token totals, whether the turn is still working or waiting on you, a pre-built lowercase blob for fast search, and the untouched original record for the Raw tab.

Loading is separate from adapting: src/lib/loadData.ts just fetches the feed files and sanity-checks them; mergeSessions() then interleaves all three into one list, newest first.

3Cycles — the unit you actually read

A raw session is a flat stream of hundreds of events. Karin regroups it into cycles, where one cycle = one thing you asked for, plus all the work the AI did because of it. That's the shape the detail page renders.

In src/lib/unifiedCycles.ts, events from any source are flattened into a common UnifiedEntry (a message, a thought, a tool call, a file edit, a usage frame…), then a new cycle is opened whenever you typed a prompt — or answered a question the AI asked you. Each cycle also derives its own prompt label, counts, model, and timing, split into time actually working versus time waiting on you.

Who spent the tokens

Tools report token usage at different granularity, so attributeCycleUsage() behaves differently per source — this is the one place the sources genuinely differ:

  • Claude reports usage per reply, so bars are measured and sit exactly where they belong.
  • Codex reports usage in periodic frames, so Karin splits each frame across the cards it covered, weighted by their text length — these bars are honest estimates and labelled as such.
  • Warp reports one running total per model and nothing per event, so it gets no per-cycle bars at all.

4Turning tokens into a number you care about

src/lib/pricing.ts can express the same usage three ways, toggled in the UI:

  • tokens — the raw counts.
  • token units — everything restated as "how many of this kind of token would that cost", since input, cached and output tokens are priced very differently.
  • money — real currency, in USD or EUR, whole or cents.

Money has a second switch: API basis uses published list prices; subscription basis divides those by a per-tool divisor to estimate what the work actually costs you on a flat monthly plan. Rate tables for each model live in that same file, and the reference table shown in the UI is derived from them so the two can't disagree.

Warp is deliberately not priced. It only records a cumulative token count per model with no input/output split, and those two bill at wildly different rates — any cost figure would be invented. The rate lookup returns nothing for Warp and the header says "Not priced". This is a decision, not a gap.

5What's on screen

PieceShows
App.tsxBoots the data, then picks the view: sessions, timeline, summary, or v2.
Sidebar.tsxSearchable, filterable session list with per-session usage bars, plus the global unit/price toggles.
SessionDetail.tsxOne session: header, meta, pricing panel, then its cycles.
Cycle.tsx / EventEntry.tsxA single turn, and each individual event card inside it.
TimelinePage.tsxAll sessions laid out across the day as bars.
SummaryPage.tsx"What happened" — reconstructed work items and where effort went per project.

State lives in one Zustand store, src/store/karin.ts: the three feeds, the derived session list, your selection, search, and every toggle (all persisted to local storage). boot() loads everything in parallel, then a refresh loop re-fetches the feeds every 5 seconds and only re-renders if something is genuinely newer. Sidebar order is frozen for a few minutes so rows don't jump around while you're reading.

6Running it

CommandWhat you get
./karin.ps1Local deploy on :4173 — a real built bundle with your data baked in. Source changes need a rebuild.
./karin.ps1 -DevDev server on :5173 — hot reload, best for iterating.
./karin.ps1 -TunnelAdds a Cloudflare URL so you can open it from your phone. Still served from this PC.

Either way the launcher also starts the indexers in watch mode, so the feeds keep updating underneath. Two footguns worth knowing: a new feed file must be added to the DATA_FILES allowlist in vite.config.ts or it never reaches the built page; and committing does not update :4173 — that URL serves the built folder, so it needs a rebuild (a post-commit hook now does this automatically).

7Why it's local-only

There used to be a public copy on GitHub Pages. It was removed on purpose: transcripts are sensitive and publishing them would defeat the point, the dataset is too big for a static host, and a static host can't see live activity on your machine anyway. The tunnel exists to reach this machine remotely, not to copy data anywhere.
